# 小学生朗读激励抽奖应用 - 简要开发文档

## 1. 项目目标

开发一个本地 Windows 桌面应用，用于通过麦克风检测学生朗读音量，驱动"抽奖结果逐渐揭晓"的互动效果，以提升小学生阅读兴趣。

目标特点：
- 本地运行，无需联网
- 适配较低版本 Windows
- 尽量免安装、直接运行
- 操作简单，适合教师使用
- 两层界面：配置页 -> 活动页

---

## 2. 核心需求

### 2.1 业务流程

1. 打开程序，进入**一层配置界面**
2. 教师选择麦克风、设置阈值、编辑奖项及概率
3. 点击"开始"，进入**二层活动界面**
4. 系统预先抽取本轮结果，并以模糊/遮罩状态显示
5. 学生朗读时，若音量超过阈值，则结果逐渐清晰
6. 若中途低于阈值，则清晰进度缓慢回退
7. 连续达到目标时长（默认 5 秒）后，结果完全显示并固定
8. 教师可重置并开始下一轮

---

## 3. 技术选型（采用 Python 路线）

### 3.1 选型结论

采用：

- **语言**：Python 3.10/3.11
- **GUI**：PySide6
- **音频采集**：sounddevice / pyaudio（二选一，优先 sounddevice）
- **配置存储**：JSON
- **打包**：PyInstaller

### 3.2 选型理由

在"开发效率、界面能力、模块清晰、后期维护"之间权衡后，采用 Python 路线：
- 开发速度快，适合快速迭代
- 模块拆分清晰，便于分 phase 开发
- PySide6 足以完成双界面与基础动画
- 可打包为 exe，满足教师直接双击运行的目标

### 3.3 风险与说明

Python 不是 Windows 原生技术，打包后体积会比原生程序大一些；但在当前需求规模下，仍是可接受且高性价比方案。  
为兼顾稳定性，建议交付形式为：
- `exe + 配置文件 + 必要依赖` 的免安装目录版  
而不是强求极致单文件。

---

## 4. 系统架构

### 4.1 总体结构

采用**双界面 + 模块化分层架构**：

- **UI层**
  - 配置界面
  - 活动界面
- **服务层**
  - 音频采集服务
  - 抽奖服务
  - 配置管理服务
  - 动画/状态推进服务
- **数据层**
  - JSON 配置文件
  - 奖项数据模型
  - 运行时状态模型

### 4.2 模块划分

#### A. 配置界面模块

功能：
- 枚举本机麦克风
- 选择麦克风设备
- 设置音量阈值
- 设置揭晓时长/回退速度
- 添加、删除、编辑奖项
- 设置奖项权重
- 点击开始进入活动界面

#### B. 活动界面模块

功能：
- 展示当前抽奖结果区域
- 显示模糊/遮罩/透明度变化效果
- 展示实时音量条与揭晓进度
- 达成条件后锁定结果
- 支持"下一轮 / 重置"

#### C. 音频采集模块

功能：
- 获取麦克风输入
- 实时计算音量值
- 进行平滑处理
- 输出统一音量强度

#### D. 抽奖模块

功能：
- 根据权重进行随机抽奖
- 每轮开始前预抽一个结果
- 提供结果锁定与重置

#### E. 配置模块

功能：
- 读取/保存 JSON 配置
- 校验奖项权重与参数合法性
- 提供默认配置

#### F. 状态控制模块

功能：
- 根据音量阈值推进 reveal progress
- 超阈值时前进，低阈值时回退
- 达到满值后触发结果固定

## 5. 界面设计

### 5.1 一层界面：教师配置页

建议包含：

- 麦克风测试音量条
- 音量阈值设置

- 回退时间设置（如 6~8 秒）
- 奖项编辑表格：
  - 奖项名称
  - 权重
  - 删除按钮
- 添加奖项按钮
- 保存配置按钮
- 开始按钮

### 5.2 二层界面：活动页

建议包含：
- 中央大区域：抽奖结果显示
- 模糊/半透明/遮罩揭晓效果
- 当前音量条
- 当前揭晓进度条
- 结果固定后显示提示
- 下一轮按钮
- 返回配置页按钮（可选）

## 6. 核心交互逻辑

### 6.1 结果揭晓逻辑

- 每轮开始时，先按权重抽取 1 个结果
- 初始状态结果不可清晰辨认
- 若音量高于阈值：
  - progress 按时间增长
- 若音量低于阈值：
  - progress 按较慢速度下降
- 当 progress >= 100%：
  - 结果清晰显示
  - 本轮锁定，不再回退

### 6.2 音量处理建议

为避免抖动：
- 对音量做移动平均或指数平滑
- 增加短暂缓冲区，避免瞬时掉落导致频繁回退
- 前进速度与回退速度可分开配置

## 9. 分 Phase 开发计划

**要求**：每个 phase 增量代码尽量控制在约 2000 行以内

### Phase 1：基础骨架与配置页

**目标**：
- 搭建项目结构
- 完成配置页 UI
- 完成配置文件读写
- 支持奖项增删改查
- 支持麦克风列表展示（先不做实时采集）

**输出**：
- 可运行的一层界面
- 可保存/加载配置

**建议控制**：
- 约 800 ~ 1500 行

### Phase 2：音频采集与阈值检测

**目标**：
- 接入麦克风实时采集
- 配置页显示实时音量条
- 实现阈值测试
- 完成音量平滑处理

**输出**：
- 可选择麦克风并看到音量反馈
- 可验证阈值是否合理

**建议控制**：
- 增量约 1000 ~ 1800 行

### Phase 3：二层活动页与抽奖逻辑

**目标**：
- 点击"开始"进入活动页
- 实现加权抽奖
- 每轮开始时预抽结果
- 活动页展示结果占位、音量条、进度条
- 支持下一轮/重置

**输出**：
- 完整双界面闭环
- 完整抽奖流程可运行

**建议控制**：
- 增量约 1200 ~ 2000 行

### Phase 4：揭晓动画与状态推进

**目标**：
- 将音量值映射到 reveal progress
- 完成"前进/回退"逻辑
- 实现模糊、透明度或遮罩揭晓效果
- 达成 100% 后锁定结果

**输出**：
- 核心互动体验完成

**建议控制**：
- 增量约 1000 ~ 1800 行

### Phase 5：稳定性、打包与教师可用性优化

**目标**：
- 异常处理
- 默认配置兜底
- 参数合法性校验
- 简化操作文案
- PyInstaller 打包
- 在目标 Windows 机器测试

**输出**：
- 可交付版本
- 免安装目录版 exe

**建议控制**：
- 增量约 800 ~ 1500 行

## 10. 关键实现原则

- 优先保证稳定可用，再优化视觉效果
- 先完成最小闭环，再逐步增加动画细节
- 奖项与参数全部可配置，不把业务写死
- 所有运行参数尽量落盘保存，降低教师重复配置成本
- UI 操作尽量少步骤，避免复杂技术概念暴露给使用者
